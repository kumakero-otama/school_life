現在の GitHub Pages ベースの構成は、静的サイト上でブラウザ側 JavaScript による簡易パスワード制御を行っています。ここでは問題点と、現実的な改善案をチェックリストとして残します。

### **1. 現在の問題点**

| 購入 | 準備 | 分類 | 内容 | 備考 |
| :---: | :---: | :--- | :--- | :--- |
| [ ] | [ ] | 問題 | パスワード判定ロジックがクライアント側にある | `assets/app.js` を見れば認証方式が分かる。 |
| [ ] | [ ] | 問題 | ハッシュ値が公開配信される | 厳密には秘密情報ではなく、総当たりや模倣の対象になる。 |
| [ ] | [ ] | 問題 | `sessionStorage` を直接書き換えると通過できる | ブラウザ開発者ツールで改変可能。 |
| [ ] | [ ] | 問題 | Markdown や JSON の元データ自体が公開URLで取得できる | ページを隠してもデータ配信は止められない。 |
| [ ] | [ ] | 問題 | GitHub Pages だけではサーバー側認証を置けない | 本格的なアクセス制御には向かない。 |

### **2. 当面の改善候補**

| 購入 | 準備 | 分類 | 内容 | 備考 |

| [ ] | [ ] | 改善 | 見られて困る情報を Pages 側に置かない | 個人情報や機微情報は除外。 |
